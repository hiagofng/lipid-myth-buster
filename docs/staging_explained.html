Staging Layer Explained

Everything your stg_cac_health_screening.sql does — the SQL, the medical science, and why it matters for LipidMythBuster.

You are here: Staging

What is the Staging Layer?

The staging layer sits between raw data (ingestion) and final analytics (reports/dashboard). It's where you clean, enrich, and derive new columns without modifying the original data.

YOUR PIPELINE (3 layers): INGESTION STAGING REPORTS Raw data as-is --> Clean + Enrich --> Aggregate for dashboard 1,688 rows Still 1,688 rows Summarized rows 22 columns ~30 columns (new derived) Grouped metrics No business logic Clinical categories added Dashboard-ready
💡
Key principle: The row count doesn't change. Staging adds columns, not rows. You're enriching each patient's record with derived clinical categories.

The Kitchen Analogy

Analogy

Think of your data pipeline like a restaurant kitchen:

Ingestion = Delivery Truck

Raw ingredients arrive from the supplier (Figshare). You don't wash, cut, or season them. You just put them in the fridge (DuckDB) exactly as they came.

That's your ingestion.cac_health_screening table.

Staging = Prep Kitchen

A chef takes ingredients from the fridge and preps them: washes vegetables, cuts meat into portions, makes sauces. The raw ingredients still exist in the fridge untouched.

That's your staging.stg_cac_health_screening — it reads from raw and creates enriched versions.

Reports = Plated Dish

The plated dish goes to the customer (your dashboard). It's the final presentation — aggregated, beautiful, ready to consume.

That's your reports layer (coming next).

Why not prep in the delivery truck?

If your prep logic has a bug (wrong seasoning), you just redo the prep. You don't need to call the supplier again. Separation of concerns = if staging SQL is wrong, re-run staging. Ingestion (the expensive download) stays untouched.

Pipeline Flow & DAG

The depends: keyword in the Bruin header creates a DAG:

Bruin reads your asset headers and builds this execution order: ingestion.cac_health_screening (Python asset — runs FIRST) | | depends: v staging.stg_cac_health_screening (SQL asset — runs SECOND) | | depends: v reports.??? (coming next — runs THIRD)

When you run bruin run bruin/pipeline, Bruin looks at all depends: declarations and runs assets in the right order. You never need to say "run ingestion first" — the DAG handles it.

The CTE Pattern

Your SQL uses a CTE called base. Here's why:

Analogy

Imagine you're grading 1,688 exams. For each exam you need to check: "Did they pass math? Did they pass science? Did they pass English?"

Without CTE: You check each subject 3 separate times for every calculation that needs it. That's redundant.

With CTE: You make ONE pass through all exams, stamp "pass/fail" for each subject, THEN use those stamps for all your calculations. One pass, reuse everywhere.

WITH base AS (
    SELECT
        *,  -- keep ALL original columns

        -- Compute flags ONCE here...
        CASE WHEN tg >= 150 THEN 1 ELSE 0 END AS _flag_high_tg,
        -- ...so we can reuse them below without repeating logic

    FROM ingestion.cac_health_screening
    WHERE no_figshare IS NOT NULL
)

SELECT
    -- Now _flag_high_tg is available as a column
    _flag_high_tg AS high_triglycerides,                 -- used here
    _flag_high_tg + _flag_high_glucose + ... AS score   -- AND here
FROM base

The _ prefix on _flag_high_tg is a convention meaning "internal/temporary — not meant for the final output." The final SELECT renames them to clean names like high_triglycerides.

Section 2: CAC Risk Category

What is CACS?

The Coronary Artery Calcium Score (CACS) is measured via a CT scan of the heart. Calcium deposits in the arteries indicate atherosclerosis (plaque buildup).

Analogy

Imagine your arteries are water pipes. Over time, mineral deposits (calcium) build up inside. A CACS scan is like an X-ray of your pipes — it counts how much buildup there is. Zero = clean pipes. 400+ = heavily clogged.

The Clinical Cutoffs

0 = None
1-99 = Mild
100-399 = Moderate
400+ = Severe

Source: Multi-Ethnic Study of Atherosclerosis (MESA) guidelines

The SQL

CASE
    WHEN cacs = 0                  THEN 'No calcification'
    WHEN cacs BETWEEN 1 AND 99     THEN 'Mild'
    WHEN cacs BETWEEN 100 AND 399  THEN 'Moderate'
    WHEN cacs >= 400               THEN 'Severe'
END AS cac_risk_category

CASE WHEN is SQL's version of if/else. It checks conditions top to bottom and returns the first match:

-- Pseudocode equivalent:
if cacs == 0:
    return "No calcification"
elif cacs between 1 and 99:
    return "Mild"
elif cacs between 100 and 399:
    return "Moderate"
elif cacs >= 400:
    return "Severe"
else:
    return NULL  -- no ELSE clause = NULL if nothing matches

The result gets stored in a new column called cac_risk_category (the AS keyword names it).

Why this matters for your thesis

CACS is the outcome variable — the thing you're trying to predict/explain. Your thesis argues that metabolic health (triglycerides, glucose, etc.) predicts CACS better than LDL alone. This category makes it easy to visualize on a dashboard: "What % of people with 'Severe' CAC had high LDL vs high metabolic risk?"

Section 3: LDL Cholesterol Category

What is LDL?

LDL (Low-Density Lipoprotein) carries cholesterol through your bloodstream. It's famously called "bad cholesterol" because the conventional view is: more LDL = more plaque = more heart disease.

Analogy

Think of LDL as delivery trucks carrying cholesterol packages through your blood vessels (highways). The conventional view says: "More trucks on the highway = more traffic jams (plaque)." Your project questions this: "Maybe it's not the NUMBER of trucks that matters, but whether the HIGHWAY is damaged (metabolic health) in the first place."

The ATP III Classification

These cutoffs come from the NCEP ATP III guidelines:

<100 Optimal
100-129 Near
130-159 Border
160-189 High
190+ V.High
CASE
    WHEN ldl_chol < 100  THEN 'Optimal'
    WHEN ldl_chol < 130  THEN 'Near optimal'
    WHEN ldl_chol < 160  THEN 'Borderline high'
    WHEN ldl_chol < 190  THEN 'High'
    WHEN ldl_chol >= 190 THEN 'Very high'
END AS ldl_category

Using < (less than) creates non-overlapping ranges naturally:

  • If LDL = 129.5, it's < 130 so "Near optimal" ✔
  • If LDL = 130.0, it fails < 130, moves to next: < 160 so "Borderline high" ✔

With BETWEEN, you'd need to worry about edge cases with decimals (does 129.5 fall in 100-129 or 130-159?). The < approach is cleaner for continuous (float) values.

Section 4: Metabolic Risk Flags

These five binary flags are the heart of your thesis. Each one captures a different dimension of metabolic health:

high_triglycerides

TG >= 150 mg/dL

Triglycerides (TG) are blood fats. High TG often signals insulin resistance — the body is storing too much energy as fat.

Threshold: 150 mg/dL

high_glucose

Glucose >= 100 mg/dL

Fasting blood glucose above 100 indicates impaired fasting glucose — the first warning sign of diabetes.

Threshold: 100 mg/dL

low_hdl

M: <40 | F: <50 mg/dL

HDL is the "garbage truck" that removes cholesterol from arteries. Low HDL means less cleanup. The threshold differs by sex because women naturally have higher HDL.

Sex-specific threshold

obese

BMI >= 30

BMI >= 30 is clinical obesity. Used here as a proxy for waist circumference (which isn't in the dataset but is the standard MetS criterion).

Proxy measure

prediabetic_or_diabetic

HbA1c >= 5.7%

HbA1c is the "3-month blood sugar report card." Unlike fasting glucose (a single snapshot), HbA1c shows the trend. >= 5.7% = prediabetes territory.

Strong predictor

The SQL for sex-specific HDL threshold

CASE
    WHEN sex = 'M' AND hdl_chol < 40 THEN 1
    WHEN sex = 'F' AND hdl_chol < 50 THEN 1
    ELSE 0
END AS _flag_low_hdl

This checks two conditions: the sex AND the HDL value. A male with HDL = 42 gets 0 (normal), but a female with HDL = 42 gets 1 (low) because women's threshold is higher.

Section 5: Smoking Status

The raw data has two separate binary columns. Staging merges them into one clean categorical column:

RAW (two columns): STAGED (one column): ex_smoker | current_smoker smoking_status ---------+----------------- ---------------- 0 | 1 --> "Current" 1 | 0 --> "Former" 0 | 0 --> "Never" 1 | 1 --> "Current" (current takes priority)
CASE
    WHEN current_smoker = 1 THEN 'Current'    -- checked FIRST (priority)
    WHEN ex_smoker = 1      THEN 'Former'
    ELSE 'Never'
END AS smoking_status

Order matters! current_smoker is checked first. If someone has both flags = 1 (data quirk), they're classified as "Current" because that's the more relevant health risk.

Section 6: TG/HDL Ratio & LDL Particle Pattern

This is one of the most powerful additions you made. Here's the science:

Not all LDL is created equal

Analogy

Imagine LDL particles as balls being thrown at a wall (your artery). Pattern A particles are like big beach balls — they bounce off harmlessly. Pattern B particles are like golf balls — small, dense, and they punch holes in the wall (damage the artery lining).

Two people can have the SAME total LDL number, but one has mostly beach balls (safe) and the other has mostly golf balls (dangerous). This is why total LDL alone is misleading.

The TG/HDL ratio as a proxy

Directly measuring LDL particle size requires an expensive NMR test. But research (Maruyama 2003, Boizel 2000) found that the TG/HDL ratio is a reliable proxy:

Triglycerides (TG) HDL Cholesterol   =   TG/HDL Ratio If ratio > 2.5 → likely small dense LDL (Pattern B, dangerous) If ratio ≤ 2.5 → likely large buoyant LDL (Pattern A, safer)

The SQL

-- The ratio itself (numeric)
ROUND(tg / NULLIF(hdl_chol, 0), 2) AS tg_hdl_ratio,

-- The pattern classification (categorical)
CASE
    WHEN (tg / NULLIF(hdl_chol, 0)) > 2.5
        THEN 'Pattern B (Small Dense)'
    WHEN (tg / NULLIF(hdl_chol, 0)) IS NOT NULL
        THEN 'Pattern A (Large Buoyant)'
    ELSE NULL
END AS estimated_ldl_particle_pattern

NULLIF(hdl_chol, 0) says: "If hdl_chol equals 0, return NULL instead."

Why? Because dividing by zero crashes your query. If someone has HDL = 0 (unlikely but possible in dirty data), tg / 0 = error. But tg / NULL = NULL (safely handled).

-- Without NULLIF:
150 / 0    = ERROR! Division by zero!

-- With NULLIF:
150 / NULL = NULL  (safe — just means "unknown")

ROUND(value, 2) rounds to 2 decimal places:

150 / 48 = 3.125      → ROUND = 3.13
120 / 55 = 2.18181... → ROUND = 2.18

Makes the numbers cleaner for dashboard display. No one needs 15 decimal places of TG/HDL ratio.

Why this matters for your thesis

If your dashboard shows that people with Pattern B (Small Dense) LDL have worse CAC scores regardless of total LDL level, that's powerful evidence. It says: "It's not about HOW MUCH LDL you have, it's about WHAT KIND."

Section 7: Non-HDL Cholesterol

Non-HDL Cholesterol  =  Total Cholesterol  -  HDL Cholesterol Captures ALL atherogenic (artery-damaging) lipoproteins: LDL + VLDL + IDL 
(chol - hdl_chol) AS non_hdl_chol

Why not just use LDL?

Analogy

Imagine you're counting all the "bad guys" in a room. LDL only counts the ones wearing red shirts. Non-HDL counts everyone who isn't a good guy (HDL) — red shirts, blue shirts, gray shirts. It's a more complete picture of threat.

The AHA/ACC now recommends non-HDL over LDL alone because:

  • Standard LDL is often calculated (Friedewald equation), not measured directly
  • The calculation becomes inaccurate when triglycerides are high (>400 mg/dL)
  • Non-HDL captures VLDL and other atherogenic particles that LDL-C misses

Section 8: Metabolic Syndrome Score (0-5)

Metabolic Syndrome is diagnosed when someone has 3 or more of 5 criteria. Your score counts how many they have:

The 5 MetS criteria (NCEP ATP III, adapted): Flag Threshold What it means ──────────────────── ────────────────────── ────────────────────────── _flag_high_tg TG >= 150 mg/dL Too much fat in blood _flag_high_glucose Glucose >= 100 mg/dL Blood sugar too high _flag_low_hdl HDL < 40(M) / 50(F) Not enough "good" cholesterol _flag_obese BMI >= 30 Obesity (proxy for waist circ.) _flag_hypertension hypertension = 1 High blood pressure Score = sum of all flags (0 to 5) Score >= 3 = Clinical Metabolic Syndrome diagnosis
-- Simple addition of the five 0/1 flags
(_flag_high_tg + _flag_high_glucose + _flag_low_hdl + _flag_obese + _flag_hypertension)
    AS metabolic_syndrome_score
Analogy

Think of it like a "health violation score" for a restaurant. Each violation (dirty kitchen, expired food, no hand-washing, pest problem, wrong temperature) is 1 point. Score 0 = spotless. Score 3+ = you get shut down. Same idea — each metabolic flag is one violation, and 3+ means your metabolism is in serious trouble.

⚠️
Limitation: The real MetS criteria use waist circumference, not BMI. Our dataset doesn't have waist measurements, so BMI >= 30 is used as a proxy. Mention this in your project README — reviewers appreciate intellectual honesty.

Section 9: Metabolic Health Status

This is the most important derived column for your thesis. It creates a binary grouping:

Score = 0 → "Metabolically healthy" Zero flags triggered Score >= 1 → "Metabolically unhealthy" At least one flag triggered
CASE
    WHEN (_flag_high_tg + _flag_high_glucose + _flag_low_hdl
          + _flag_obese + _flag_hypertension) = 0
        THEN 'Metabolically healthy'
    ELSE 'Metabolically unhealthy'
END AS metabolic_health_status

Why this column is the key to your thesis

Quality Checks (Bruin Header)

The checks: in the Bruin header run after the SQL executes. If any check fails, the pipeline stops and reports the error.

CheckWhat it doesApplied to
not_null Fails if ANY row has NULL in this column no_figshare, cacs, cac_risk_category, ldl_category, metabolic_health_status
unique Fails if any value appears more than once no_figshare (each subject should be unique)
accepted_values Fails if any value is NOT in the allowed list cac_risk_category, ldl_category, smoking_status, estimated_ldl_particle_pattern, metabolic_health_status

The original draft had positive on both cacs and metabolic_syndrome_score. The problem:

-- "positive" means > 0
cacs = 0          → FAILS positive check!  But 0 is valid ("No calcification")
met_score = 0     → FAILS positive check!  But 0 is valid ("Metabolically healthy")

Zero is a meaningful, valid value in both cases. The check would have rejected perfectly good data. We removed it.
